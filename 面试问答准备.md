# AI 创作平台 — 面试问答准备

## Q1: 为什么用 LangGraph 而不是 LangChain 原生 Agent？StateGraph 怎么设计状态流转？

LangChain 自带的 AgentExecutor 就是一个死循环——AI 调工具、看结果、再想、再调工具，来回转。但我们的场景不是循环，是一条流水线：先准备素材，再写文章，然后审校和一致性检查同时跑，最后交给用户审批。

AgentExecutor 有两个硬伤：第一，它只能串行执行，写完之后审校和一致性检查没法同时跑，只能一个接一个；第二，它没有暂停机制，我们需要在某个节点停下来等用户说"行不行"，它做不到，得自己硬写暂停恢复的逻辑。

LangGraph 的 StateGraph 就是为这种流水线设计的。并行很简单，给 writer 画两条出边就行，一条去审校一条去一致性检查，LangGraph 自动让它们同时跑。暂停也简单，设一个 interrupt_before 参数，图跑到审批节点就停，用户点了"接受"再恢复继续跑，状态还能存下来不怕丢。

状态流转的设计就是所有节点共用一个大字典，每个节点只改自己负责的那几个字段。比如 writer 改 draft，审校改 critiques，一致性检查改 report，大家互不干扰。critiques 这个字段用了 reducer，意思是每次审校不是覆盖而是追加，这样多轮审校的意见都能保留下来。

## Q2: 审校节点并行执行，效率提升 50%，具体怎么实现？节点间怎么通信、怎么合并？

就是在画图的时候，writer 后面连了两个节点，审校和一致性检查。LangGraph 看到一个节点后面有两个下游，就自动让它们同时跑，不用我手动写并发逻辑。

两个节点怎么通信？它们共享同一个状态对象。writer 把文章写好放到 draft 里，审校读 draft 然后把意见放到 critiques 里，一致性检查也读 draft 然后把报告放到另一个字段里。两个节点写的是不同的字段，不会打架。等两个都跑完了，LangGraph 自动把结果合并到一起，然后往下推到审批节点。

为什么说省 50%？审校和一致性检查都是调一次大模型，各花三四秒。串行的话加起来七八秒，并行的话就看谁慢，大概三四秒就完了，差不多省一半时间。

## Q3: RAG 双路径检索，什么场景用 ID？什么场景用向量？怎么保证不冲突不重复？

这个设计是给用户两种选择：第一种，用户在弹窗里主动勾选了"我要用这个角色、这个世界观设定"，那我们就按 ID 直接从数据库查出来，全文给 AI，一字不漏。因为用户明确选了，就得完整给到。

第二种，用户没选任何素材，那就让 AI 自己判断该看什么。把当前章节的内容做成向量，去素材库里找最相关的世界观和角色推荐过来。

两条路不会冲突，因为它们是 if-else 互斥的。有选中 ID 就走第一条路，没有就走第二条路，同一时刻只走一条。第一条路完全不碰向量检索，所以不会有重复结果混进来。

举个例子：用户写第五章，主动选了"主角"和"魔法体系"，就走精确路径，把这两个的完整设定塞给 AI。用户懒得选，AI 就看第五章写了什么，自动找相关的世界观和角色推荐。

## Q4: 为什么选 pgvector 而不是 Chroma/FAISS？向量维度、索引类型、检索策略是什么？

选 pgvector 就是图两件事：数据一致和运维简单。

数据一致——我们的世界观和角色本来就存在 PostgreSQL 里，向量直接作为同一张表的一个字段存着。素材删了，向量也跟着没了，因为都在一个事务里。如果用 Chroma 或 FAISS，向量在另一个系统里，素材删了还得单独去向量库删，任何一步漏了就数据不一致。

运维简单——一个 PostgreSQL 就够了，不用再部署一个 Chroma 服务或者管理 FAISS 的索引文件。

现阶段数据量很小，每个项目就几十到几百条素材，检索就是从数据库拉出来在 Python 里算余弦相似度排序，够用了。以后数据多了，建个 pgvector 的 IVFFlat 索引，改成在数据库里直接算距离，代码改动很小。

Chroma 和 FAISS 是百万级向量、毫秒级检索才需要的，我们暂时用不上。

## Q5: SSE 流式输出怎么做到 Token 级推送？怎么处理断连、重连、内容错乱？

怎么做到逐字推送？LangGraph 有个 astream_events 接口，大模型每吐出一个字就触发一个事件，我抓住这个事件，把这一个字通过 SSE 推给前端，前端收到就立刻显示在编辑器里。用户看到的效果就是 AI 在逐字打字，像真人一样。

但有个细节：不是所有节点都要逐字推送。只有写作节点需要，审校和一致性检查是一次性给完整结果就行。所以我用一个变量记着"当前在流式输出的是哪个节点"，只有确认是写作节点的时候才把每个字推出去，审校节点的输出不会被误推成写作内容。

断连怎么办？前端有个取消按钮，点了之后请求直接中断，后端的生成过程也自动停了，不会继续浪费大模型的调用。续写模式下已经推到编辑器里的字不会丢，用户重新点按钮就行。我还做了串行控制，同一时间只允许一个 SSE 连接，新请求会自动取消旧的，这样不会出现两段内容混在一起的情况。

## Q6: LLM Provider 抽象层怎么设计？怎么兼容多厂商大模型？API Key 怎么加密存储？

怎么设计？就是一个抽象基类定义了两个方法：生成文本和流式生成文本。然后两个实现类，一个是 Mock 的用于测试，一个是 OpenAI 的用于生产。

怎么兼容多厂商？核心思路是利用 OpenAI 兼容协议。现在国内主流厂商——DeepSeek、通义千问、智谱、Moonshot 这些——都做了 OpenAI 兼容的接口，只是地址不同。所以底层统一用 OpenAI 的客户端，用户只需要告诉我三个东西：选哪个厂商、API Key 是什么、用哪个模型。我在后端维护了一张厂商地址的对照表，想加一个新厂商就加一行地址，其他代码不用改。

API Key 怎么存？用户提交 Key 的时候，我用 Fernet 加密算法加密成一段乱码存到数据库，数据库里看不到明文。真正要用的时候，从数据库读出来解密再传给大模型。对外接口只返回"有没有设置 Key"这个布尔值，永远不返回密文或明文。

不过这个方案有个局限：加密密钥是从环境变量派生的，如果环境变量泄露了，API Key 也保不住。生产环境应该用云厂商的 KMS 服务来管密钥，这是我们后面要改进的地方。

## Q7: RAG 在这个项目里是怎么应用的？

RAG 在我们项目里的作用就是**给 AI 提供创作上下文**。AI 写小说不能凭空编，它得知道这个世界的设定是什么、有哪些角色、前面写了什么剧情，不然写出来的东西跟已有内容对不上。

具体怎么做的？用户创建项目的时候会录入三类素材：大纲（每章要写什么）、角色（谁是谁、什么性格）、世界观（这个世界的规则和设定）。这些素材存到数据库之后，我们会用 embedding 模型把它们转成向量，也存到数据库里。

当 AI 要写新章节的时候，context_loader 节点会把这些素材组装成一段上下文文字，塞到给大模型的 prompt 里。这样 AI 就知道"哦，这个角色是反派、这个世界有魔法体系、上一章主角刚遇到了危险"，写出来的内容就能跟已有设定对上。

这里有个关键设计就是双路径。用户主动选了素材就走精确加载，把选中的素材全文塞进去，一个字不漏。用户没选就走向量搜索，用当前章节内容做 query，去素材库里找最相关的推荐过来。两种方式互斥，不会冲突。

还有一个细节是前文加载。不管走哪条路径，我们都会把最近三章的内容截取前 500 字放进上下文，这样 AI 就知道前面剧情发展到哪了，续写的时候能接上。

简单说，RAG 在我们这里不是做问答检索，而是做**创作素材的上下文注入**，让 AI 的输出跟项目的世界观和剧情保持一致。

## Q8: LangGraph 在这个项目里是怎么应用的？

LangGraph 在我们项目里就是**编排 AI 写作的流水线**。整个生成过程不是一次 LLM 调用就完了，而是多个 AI 角色接力完成，每个角色干自己的事，LangGraph 负责调度谁先谁后、谁和谁并行、什么时候停下来等用户。

具体来说，我们定义了一个状态图，有五个节点：

第一个是 context_loader，负责把素材和前文组装成上下文。第二个是 writer，拿到上下文开始写文章。第三个是 critic，审校 writer 写的内容，挑毛病。第四个是 consistency_checker，检查写出来的东西跟世界观设定有没有矛盾。第五个是 human_review，暂停等用户审批。

writer 跑完之后，critic 和 consistency_checker 同时启动，这是 LangGraph 的并行边实现的，不用我手动写并发。两个都跑完之后汇到 human_review，用 interrupt_before 让整个图暂停，等用户说"接受"还是"拒绝"。如果用户拒绝想改，就回到 writer 根据审校意见重写，revision_count 加一，最多迭代三轮。

除了默认流水线，我们还做了**动态图构建**。用户可以在界面上配置自定义的 AI 专家，比如加一个"氛围描写专家"放在 writer 后面。系统会根据专家的 workflow_position 字段动态往图里插节点和边，replace_writer 就替换掉默认的 writer，pre_writer 就在 writer 前面加一步。这样不懂代码的用户也能自己搭 AI 团队。

状态流转用的是 TypedDict，所有节点共享一个状态字典。每个节点只改自己负责的字段，比如 writer 改 draft、critic 改 critiques，互不干扰。critiques 用了 reducer，意思是追加不是覆盖，多轮审校的意见都能累积下来。

还有一个点是 checkpoint。我们用 MemorySaver 做持久化，图暂停的时候状态存下来，用户审批后从断点恢复继续跑，不会丢失中间结果。

简单说，LangGraph 在我们这里解决的核心问题就是：**把多个 AI 角色的协作流程从硬编码的函数调用，变成了可配置、可并行、可暂停的状态图**。

## Q9: 项目有断点重续功能吗？怎么实现的？

有，而且是我们项目一个比较核心的设计。断点重续分两层：一层是流水线级别的暂停恢复，一层是状态持久化。

**流水线级别的暂停恢复：** 我们的 LangGraph 流水线有五步——素材加载、写作、审校、一致性检查、人工审批。在人工审批这个节点，我们设了 interrupt_before，意思是图跑到这里就自动暂停，不会往下走，等用户做决定。用户可以选三个动作：接受、拒绝、修订。接受就落库保存，拒绝就终止流程，修订就把用户的反馈注入状态，图从 writer 节点重新开始跑，最多迭代三轮。这个暂停不是我自己写的，是 LangGraph 原生支持的，设一个参数就行。

**状态持久化：** 图暂停的时候，所有中间状态——writer 写的草稿、critic 的审校意见、一致性检查的报告——都会通过 MemorySaver 存下来。MemorySaver 是 LangGraph 的内存级 checkpointer，它把图的完整状态快照保存起来，每个快照用 thread_id 标识。用户审批的时候，前端带着 thread_id 调 resume 接口，后端从 checkpointer 里把状态恢复出来，接着暂停的地方继续跑，之前的结果不会丢。

**具体怎么恢复的：** resume 接口收到请求后，先用 thread_id 从 checkpointer 里 aget_state 拿到当前状态，然后根据用户的选择决定下一步。如果是修订，就 aupdate_state 把反馈写进去，然后 astream_events 继续流式执行。如果是接受，就把最终内容落库到数据库。

**局限：** 现在的 MemorySaver 是内存级的，服务重启状态就没了。配置里预留了 LANGGRAPH_CHECKPOINT_DB 开关，以后可以切换到数据库持久化，这样即使服务重启也能恢复。另外 SSE 连接断了的话，已经推到前端的内容不会丢（因为编辑器是本地状态），但正在生成的内容会中断，用户需要重新触发生成。
